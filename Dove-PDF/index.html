Dove-PDF Tools Platform

A sophisticated online PDF editor and toolkit that enables users to upload, edit, highlight, annotate, merge, compress, and export PDF files directly in the browser. Built with a modern TypeScript monorepo architecture featuring advanced PDF-to-HTML conversion, real-time overlay editing, intelligent font preservation, and dual export pathways for maximum quality and flexibility.

Overview

Dove-PDF is a sophisticated online PDF editor featuring advanced PDF-to-HTML conversion with font preservation, real-time overlay editing system with normalized coordinates, and dual export pathways (direct PDF or HTML-to-PDF rendering). Built with React 19, Zustand, and IndexedDB persistence for handling large binary files.

Vision

Build a browser-based PDF editor with desktop-grade capabilities, featuring advanced conversion algorithms, intelligent font mapping, and resolution-independent overlay system.

Technical Innovation

Normalized coordinate system for overlays, custom IndexedDB serialization for binary data, 300+ font family mapping engine, and conditional export logic.

Outcome

Production-ready platform with 95%+ font preservation, pixel-perfect overlay positioning, 50MB+ PDF support in IndexedDB, and dual export pathways for quality optimization.

Problem / Challenge

Users need quick, reliable tools to manage PDF documents without installing desktop software. The core technical challenges included:

1. PDF-to-HTML Conversion: Converting complex PDFs while preserving fonts, images, layouts, and styling.
2. Interactive Editing: Enabling real-time overlay editing (highlights, annotations, signatures, images) with precise positioning.
3. Font Preservation: Handling 300+ font families from academic PDFs (LaTeX, Times New Roman, Arial variants) and mapping them to web-safe alternatives.
4. Export Quality: Maintaining original PDF quality when no edits are made, while supporting HTML-rendered exports when content is modified.
5. State Management: Persisting large binary PDF data and edit history in browser storage without performance degradation.

Solution

I architected and implemented a full-stack PDF processing platform:

Frontend: React 19 + Vite with Zustand state management and IndexedDB persistence (custom Uint8Array serialization). Normalized overlay coordinate system (0-1 ratios) for resolution-independent positioning. pdf-lib for client-side manipulation, html2canvas + jsPDF for HTML-to-PDF conversion.

Backend: Express + tRPC for type-safe APIs. Stirling PDF API integration for PDF→HTML conversion. Custom font mapping system normalizing 300+ font families. Dual export strategy: direct pdf-lib overlays OR headless Chrome HTML rendering. Multer file uploads with JSZip for asset extraction.

Conversion Pipeline: Upload → Stirling PDF API → ZIP extraction (HTML + fonts + images) → Base64 asset encoding → Font-family parsing → @font-face CSS generation → Inline style preservation → Return to frontend.

Export Process: No HTML edits = direct PDF-lib overlay application. With HTML edits = inject overlays as HTML elements → html2canvas capture → jsPDF assembly → optional Stirling compression.

Challenges & Solutions

PDF-to-HTML with Font Preservation

Problem: PDFs use embedded fonts browsers can't render. Academic PDFs use LaTeX fonts (Computer Modern, Nimbus) with obfuscated names like "GKKWXB+NimbusRomNo9L-Medi". Must map to web-safe alternatives while maintaining weight/style.

Solution: Built font mapping system: (1) Parse HTML for font-family, (2) Strip prefixes and normalize, (3) Match against 300-entry map, (4) Infer weight/style from name, (5) Generate @font-face rules with data URLs or fallbacks.

Overlay Positioning Across Zoom & Export

Problem: Users edit at various zoom levels (50-150%), but overlays must position correctly in preview, export to correct positions in final PDF, survive page reordering, work for both PDF and HTML-to-PDF rendering.

Solution: Normalized coordinate system: store positions as ratios (0-1), not pixels. Formula: normalizedX = pixelX / pageWidth. Each overlay stores pageId to survive reordering. Preview uses CSS transform: scale(), then reverse-scales mouse coords. Export multiplies by PDF/HTML dimensions.

IndexedDB Persistence with Binary Data

Problem: PDFs stored as Uint8Array (binary), but IndexedDB serialization via JSON.stringify loses binary data. 50MB+ PDFs need efficient encoding. localStorage (5MB limit) insufficient. Zustand's default serialization breaks on typed arrays.

Solution: Custom IndexedDB adapter: (1) Serialization walks object tree, detects Uint8Array, converts to base64 with markers, (2) Chunking processes in 32KB chunks, (3) Deserialization decodes base64 back to Uint8Array, (4) Migration logic for old formats, (5) Selective persistence of necessary state only.
