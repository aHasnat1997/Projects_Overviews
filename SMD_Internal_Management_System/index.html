Case Study

Sarkar Group — SMD

A production-grade internal management system for a construction and marine engineering company. Multiple roles — Super Admin, Admin, Project Manager, Engineer, Client — each operate within a tailored, permission-isolated dashboard. The platform covers project management, equipment lifecycle tracking, financial workflows, HR applications, media publishing, and real-time notifications — all in a single monolithic REST system.

Overview

Sarkar Group SMD replaces fragmented spreadsheet workflows with a unified internal platform. It handles the full operational lifecycle: registering and assigning construction equipment, managing multi-department projects with engineers and clients, processing financial requisitions and payments, handling HR applications, publishing company media, and delivering real-time notifications to all relevant users on every workflow action.

Vision

One platform for all internal operations — project assignments, equipment tracking, financial approvals, employee HR requests, and company content — with strict role-based access enforced at every layer.

Technical Innovation

Generic chainable PrismaQueryBuilder serving 15+ modules, SSE-based real-time notification service with per-user client registry, parallel Prisma transaction queries for role-specific dashboards, and a full approval workflow engine.

Outcome

Production-live system with 5 operational user roles, 20+ database models, real-time push notifications, Dockerized deployment with health checks, and all company workflows digitized.

Problem / Challenge

The company managed construction equipment, marine vessels, engineering projects, and a multi-department workforce through manual, disconnected processes.

1. Multi-role access: Different roles need isolated views and strict permission enforcement without code duplication.
2. Equipment lifecycle: Tracking equipment across STAND_BY → assigned → project completion with request/approval flows.
3. Financial workflows: Payments and requisitions needed a full submit → review → approve/reject cycle with documented decline reasons.
4. HR applications: 8 application types (leave, resignation, salary advance, loan, complaint, etc.) required structured, auditable workflows.
5. Real-time feedback: Employees needed instant notifications when admins acted on their submissions — without polling.
6. Scalable query layer: 15+ modules all need search, filter, sort, paginate, and include — duplicating this would be unmaintainable.

Solution

Backend: TypeScript + Express with a structured Rocket server class (load → initiate → launch lifecycle). JWT authentication with role-array authGuard middleware on every protected route. Generic PrismaQueryBuilder class — accepts any Prisma delegate, chainable .search() / .filter() / .sort() / .paginate() / .includeRelations() — reused across all 15+ modules. Zod validation on every mutation endpoint.

Real-time Notifications: Singleton NotificationService maintains a Map<userId, Set<Response>> for all SSE connections. On workflow events (payment/requisition/application created), it broadcasts to all admin roles. On approval/rejection, it sends a targeted notification to the submitting employee. Keepalive pings every 20s prevent proxy timeouts; connection cleanup on close/finish/error.

Dashboard Aggregation: Each role's dashboard endpoint uses Prisma $transaction to run 10–15 parallel queries — KPI counts, month-over-month delta calculations, recent records, and active project lists — in a single round trip.

Frontend: Next.js 16 App Router with role-gated route groups /(auth)/ and /(dashboard)/. Redux Toolkit + redux-persist for global state, next-auth for session management. MUI + TailwindCSS for UI. TipTap rich text editor for media posts. react-pdf + pdfjs-dist for in-app document viewing.

Challenges & Solutions

Generic Query Layer Across 15+ Modules

Problem: Every module needed search, filter, sort, paginate, and relation-include. Duplicating this in each service would mean hundreds of lines of near-identical code across 15+ services.

Solution: Built generic PrismaQueryBuilder accepting any Prisma delegate. All services instantiate it with findMany + count delegates and chain only the operations they need. One class, zero duplication.

Real-time Notifications Without WebSockets

Problem: Employees need instant feedback on submission outcomes. Adding WebSocket infrastructure (socket.io, separate server) would add significant complexity and deployment overhead.

Solution: SSE via Express res.write(). Singleton NotificationService holds a Map<userId, Set<Response>>. Events are pushed live to connected clients. Keepalive pings prevent proxy drops. Cleanup runs on close/finish/error events.

Equipment Assignment Validation Chain

Problem: Equipment can only be added to a project if the project is not Completed and the equipment is STAND_BY. The auto-approval flow must respect these rules without bypassing them.

Solution: All assignment operations go through ProjectService.addProduct which validates both conditions before writing. RequestProducts approval calls this same function internally — one validation path, no bypass possible.

What I Learned

This project deepened my understanding of designing multi-role enterprise systems where access control must be consistent across both the API and UI layers. Building the generic PrismaQueryBuilder taught me the value of abstraction — one well-designed utility eliminated hundreds of lines of repetitive query code across 15+ modules.

Implementing SSE from scratch gave me practical experience with HTTP streaming, connection lifecycle management, and the trade-offs versus WebSockets for one-directional push scenarios. The parallel dashboard aggregation with Prisma $transaction highlighted the importance of minimizing round trips for data-heavy endpoints.

The project reinforced clean module architecture — thin controllers, service-layer business logic, route-level Zod validation — as the pattern that scales best when a codebase grows to 15+ interconnected modules with complex permission requirements.
